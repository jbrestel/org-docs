Website Performance

Performance Testing revealed general poor performance of the unified database. In addition, there were several very poor performing tools. Each area of concern is listed below and proposed solutions if needed. The general strategy for fixing the website performance will be:

  1. Focus initially on worst performers.
  2. Use holistic approach to performance tuning. (tuning for a data type not a specific tool)
  3. Duplicate data as needed to speed up applications
    • Eliminate big complicated queries in the application. Create more de normalized tables
    • nosql database where appropriate (site search, record pages?)
    • flat file stores for genome browsers

A summary of the details below can be found in Google Docs

When tuning queries, we will follow the SOP defined in the Query Tuning Protocol Document. In the model-config.xml for your website, make sure to set the “slow” value of the queryMonitor Configuration so that queries you are working on will be printed to the slow query log. (path to slow query log can be found by running cattail on your dev site). 

<queryMonitor
           baseline="0.1"
           slow="0.01"
           />
